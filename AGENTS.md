# Repository Guidelines

## Project Structure & Module Organization

RowPlay Studio is a SwiftPM macOS app. Keep reusable domain logic in `Sources/RowPlayCore/` and app-only SwiftUI code in `Sources/RowPlayStudio/`. Core code covers models, analytics, replay, sync/live/connectivity, import/export/share, and support helpers. App code is split across `App/`, `Stores/`, and `Views/`. Tests live in `Tests/RowPlayCoreTests/`, with parity fixtures in `Tests/RowPlayCoreTests/Fixtures/`. Roadmap/source mapping live in `docs/`; phase specs are in `.kiro/specs/`.

## Agent Instruction Surfaces

Use this file as the shared guide for Codex, Copilot coding agent, and other agents that read `AGENTS.md`. `.github/copilot-instructions.md` is Copilot's workspace entrypoint; `CLAUDE.md` is Claude Code's entrypoint. Keep both consistent with this file. The repo also has autogenerated `.codex/environments/environment.toml`, local `.claude/settings.local.json`, and lowercase `.jules/*` notes.

## Build, Test, and Development Commands

- `swift test`: runs the XCTest suite.
- `swift test --filter WorkoutAnalyticsTests`: runs a focused test.
- `swift build`: builds the package and mirrors CI's build step.
- `./script/build_and_run.sh`: builds, stages `dist/RowPlayStudio.app`, and launches it.
- `./script/build_and_run.sh --verify`: launches the staged app and checks the process.
- `./script/build_and_run.sh --logs` or `--telemetry`: launches with live logs.

Do not launch the raw SwiftPM executable for GUI checks; use the staged bundle.

## Coding Style & Naming Conventions

Use standard Swift formatting with 4-space indentation and descriptive type names. Test files use `ThingTests.swift`; test methods use `testBehaviorUnderCondition()`. Keep SwiftUI, AppKit, and process behavior out of `RowPlayCore`; define external boundaries as protocols and inject mocks. Prefer deterministic demo data. Cache expensive library summaries in stores instead of doing O(N) derived work in views.

## Testing Guidelines

Use XCTest for core behavior. Add tests for each pure helper, storage/sync boundary, parser, replay rule, or parity port. When porting web behavior, update fixtures in `Tests/RowPlayCoreTests/Fixtures/`. Run `swift test`, `swift build`, and `git diff --check` before opening a PR; add `./script/build_and_run.sh --verify` for UI or bundle changes.

## Commit & Pull Request Guidelines

Recent history uses concise subjects such as `feat: Phase 6 - Live Mode Foundation (#9)` or `Phase 7: Hardware connectivity foundation (#10)`. Keep commits scoped to one phase or fix. PRs should explain scope, list validation commands, link the phase spec or issue, and include screenshots or notes for visible UI changes. Call out exclusions and known gaps.

## Security & Configuration Tips

Concept2 tokens must stay in Keychain-backed storage, never `UserDefaults`, files, logs, or fixtures. Use `PrivacySafeLogger` and redaction helpers for user data. Preserve demo mode without credentials.
