# Phase 11 - Production-Quality Bundled 3D Assets — Design

## Status and Design Intent

Phase 11 is a native visual-provider upgrade implemented on PR #72. The local
asset, test, architecture, staged-bundle, and bounded visual-QA gates are
recorded in the task checklist. PR #72 remains a draft until its pushed head
receives fresh GitHub CI; this document makes no merge or release claim.

The phase replaces only geometry and static scenery. Native replay timing,
course placement, articulated pivots, pose solving, cameras, effects, and
quality policy continue to define what moves and where it moves.

## Architecture

```text
script/generate_replay_assets.py
        │ deterministic ASCII USDA + provenance
        ▼
Sources/RowPlayStudio/Resources/Replay3D/
        │ Bundle.module
        ▼
ReplayAssetCatalog ─────── validates names, budgets, bounds, materials
        │
ReplayAssetLibrary ─────── loads bounded templates and returns clones
        │
ReplayRigVisualProvider
├── ReplayProceduralRigVisualProvider
└── ReplayBundledRigVisualProvider
        │ attaches visual nodes to existing logical pivots
        ▼
ReplaySportRig / ReplayAthleteRig / sport rigs
        │ native ReplayRigPoseSolver remains authoritative
        ▼
Replay3DSceneBuilder ─── ReplayEnvironmentAssetInstaller
```

`ReplayAssetCatalog` is intentionally separate from loading. It is a small,
deterministic contract layer that makes resource identity and fallback decisions
testable without scene rendering. `ReplayAssetLibrary` owns only resource
loading, validation, template caching, and clone creation. It has no replay
clock, user preference, or camera responsibility.

## Resource Generation and Packaging

The generator creates the six committed, readable USDA files in a fixed order.
It derives all geometry from project code using the Python standard library and
never fetches a model or emits opaque downloaded binary content. `--check`
regenerates in memory and fails if generated bytes or the asset contract differ
from the committed files.

Resources live under the Studio target so SwiftPM packages them with the app.
They are loaded only through `Bundle.module`; there is no file-system lookup,
network fallback, or user-configurable asset directory. `ASSET_PROVENANCE.md`
is part of the resource package and names the regeneration command and original
project ownership.

Each resource is validated before use:

1. Resolve the expected resource from the catalog and bundle.
2. Parse/load the USDA scene into an isolated template.
3. Verify the expected node set, material categories, finite bounds/transforms,
   non-empty geometry, triangle ceiling, and absence of cameras/lights.
4. Cache the immutable validated template under its sport/resource key.
5. Clone it recursively only while constructing a scene.

The validation result is all-or-nothing for a sport. A bad rig or its matching
environment does not yield an incomplete hybrid scene.

## Asset Contract

The contract identifies six resource names, their kind (`rig` or
`environment`), sport, required node names, expected bounds, material
categories, and per-file budgets. It also defines the `environment-root`,
`environment-ground`, and `environment-props` hierarchy for each environment.

Rig visual nodes deliberately match presentation roles rather than scene
ownership. For example, `visual-hand-L` is geometry attached below the native
left-hand pivot; it does not become a new moving hand controller. The existing
logical entities keep their test-visible names and receive pose transforms from
`ReplayRigPoseSolver`. The catalog is therefore the sole bridge between an
authored resource's node names and the native rig's stable pivot contract.

Material categories are semantic rather than sport-specific colour values.
Every rig uses `skin`, `hair`, `kit`, `shoe`, `accent`, `metal`, and `rubber`;
the RowErg environment uses `water`, `shore`, `foliage`, `accent`, and `metal`;
the SkiErg environment uses `snow`, `ice`, `foliage`, `accent`, and `metal`;
and the BikeErg environment uses `asphalt`, `concrete`, `accent`, `metal`, and
`foliage`. The generator gives each authored accent mesh a
`material_accent_*` name. The bundled provider uses that semantic slot name to
recolour only an independent live or rival clone, never the cached template or
unrelated skin, kit, metal, or rubber materials.

The golden fixture mirrors this metadata for deterministic tests. It is an
asset contract, not a web-parity fixture and not proof of visual acceptance.

## Visual Providers and Rig Ownership

`ReplaySportRig` selects a visual provider when its scene graph is built:

- The procedural provider creates the existing meshes and materials.
- The bundled provider requests a validated sport rig clone and attaches its
  visual children to exactly the same named pivot hierarchy.

Both providers expose the same visual attachment surface. This avoids a second
`applyPose` implementation and ensures the RowErg seat/handle/oar, SkiErg
handle/pole/cable, and BikeErg crank/pedal/wheel contact logic stays unchanged.
The procedural provider remains the low-quality path and the only fallback;
there is no part-by-part asset substitution.

Live and rival entities always use distinct recursive clones. A ghost operation
walks only the clone's material instances and converts all supported loaded
material types to the existing translucent presentation. This prevents shared
material mutation between a template, live athlete, rival, or subsequent scene.

## Environment Installation

`ReplayEnvironmentAssetInstaller` is called during scene construction, never
from a `RealityView` update closure. At bundled quality it adds the validated
sport environment and removes only the generic visual background it replaces.
It does not alter the `ReplayCourseLayout` ring, lane offsets, start/finish
marker, distance markers, camera, lights, or effects roots.

The environment is static: no new timers, simulation, water deformation, or
environment animation are introduced. Existing wakes and catch effects keep
their native semantic positions and quality-driven capacities.

## Quality, Lifecycle, and Failure Handling

Quality selection uses the existing effective quality rather than adding a new
preference:

| Effective quality | Rig visual source | Environment source |
| --- | --- | --- |
| Low | Procedural | Procedural |
| Medium | Bundled only if the complete sport set validates | Bundled only if the complete sport set validates |
| High | Bundled only if the complete sport set validates | Bundled only if the complete sport set validates |
| Ultra | Bundled only if the complete sport set validates | Bundled only if the complete sport set validates |
| Any load/contract failure | Complete procedural fallback | Complete procedural fallback |

The existing quality graph identity controls rebuilds. Rebuilding the inner
graph preserves playback state, speed, camera preset, orbit state, and rival
selection while clearing only scene-local effects according to established
rules. Template loading occurs during graph construction and never on each
frame. Failure is safe and silent to the replay workflow; a privacy-safe log,
if needed, contains only a public resource identifier.

## Accessibility, Privacy, and Performance

Assets are decorative implementation detail. The existing semantic 3D replay
element remains the sole VoiceOver surface for the scene; individual visual
node names are not exposed. No new user setting is required because visual
selection follows the existing quality tier.

Assets include no user data. They never receive a token, workout value,
filename, path, or telemetry payload. The bounded template cache prevents
repeated disk work; per-frame updates are limited to existing transforms,
visibility, opacity, and effect state. Reduced Motion continues to use stable
neutral articulation and suppress the existing transient effects.

## Test Strategy

Asset tests exercise catalog/fixture agreement, generated-resource integrity,
resource loading, node/material contracts, budgets, finite geometry, and
independent clone behavior. Failure injection verifies that absent, malformed,
or incomplete assets produce a fully functional procedural scene.

Rig tests run existing structural/contact/non-finite assertions under both low
procedural and medium bundled construction. Scene tests verify quality source
selection, environment presence only on the bundled path, state-preserving
quality rebuilds, rival material isolation, effects/camera continuity, and
reduced-motion behavior. Automated structure checks do not substitute for
staged-app visual assessment of silhouette, articulation, materials, depth, and
framing.

## Validation Discipline

The generator check, focused tests, full SwiftPM matrix, architecture scans,
staged-bundle checks, and visual QA are independent gates. The completed local
results are recorded in the task checklist and readiness documentation; the
remaining external gate is GitHub CI on the pushed PR head. The local evidence
does not establish a universal performance claim or replace the explicitly
unavailable VoiceOver, gesture, screen-size, Instruments, or GPU checks.
